## How to use this document

- Use it as an idea bank for future PRDs and backlog tickets, not as marketing copy.
- For each competitor, track: what users love, where Agent Smith is weaker, where Agent Smith can be better, and concrete product ideas to consider.
- When a competitor ships a material new capability, update this file and either link an existing ticket or create a new `Pending Debrief` ticket/PRD.

## Snapshot summary

Agent Smith's durable bet remains provider neutrality plus an open append-only session data substrate. Current competitors are moving fast on lifecycle automation, background agents, IDE-native UX, multi-agent boards, and richer extensibility. The most attractive missing opportunities for Agent Smith are:

1. **Project-intelligence map:** a repo-wide symbol/dependency map with freshness metadata and citations, inspired by Aider's repo map and IDE agents' real-time awareness.
2. **Agent workboard:** a local-first Kanban/worktree surface for managing many concurrent agents, inspired by Cline Kanban, Codex cloud/background tasks, and OpenCode multi-session.
3. **Run/verify skill generator:** a first-class way to learn how to launch and verify an app, inspired by Claude Code bundled `/run`/`/verify` skills.
4. **Shareable read-only sessions:** privacy-preserving session links/exports for debugging and team review, inspired by OpenCode share links and cloud agent task pages.
5. **Transparent background-cost controls:** budgets and attribution for autonomous/background/subagent activity, prompted by competitor usage-limit incidents and user confusion around invisible work.
6. **Persistent, checkpointed background execution with completion notifications:** OpenAI's June 2026 acquisition of Ona (formerly Gitpod) shows the market betting on agents that outlive a closed laptop; Cursor's mobile app pairs that with push notifications and remote steering. Smith's async runner explicitly deferred this (AS-054); see AS-171.
7. **Team/org governance for AI tool use:** Coder's AI Governance add-on (centralized key gateway, prompt/tool audit trail, policy enforcement) signals that team leads want one place to see what every developer's agent did, distinct from the per-run cost/secret controls Smith already plans; see AS-172.
8. **Interactive, PTY-level terminal control:** Warp's "Full Terminal Use" (open-sourced Apr 2026) drives the live terminal buffer — responding to REPL/debugger/DB-shell prompts mid-session — rather than spawn-and-capture; Smith's shell tool (AS-015) only does the latter today. See AS-186.
9. **Mining existing repo history into onboarding config, not just live-session friction:** Qodo's Custom Rules Miner derives enforceable conventions from PR/review history; Smith's living-skills loop (D7, AS-048) only learns from a session's own trial-and-error. See AS-187.
10. **Mobile/remote control is now a three-way race, not a single vendor bet:** OpenAI's Codex Remote reached GA, Cursor shipped an iOS public beta, and Sourcegraph Amp added a mobile watch/drive view — all within the same two weeks. AS-171 scoped outbound notifications only and explicitly deferred "remote steer"; AS-188 picks that question back up now that three competitors have shipped the answer.

**Trend to watch, not yet a ticket:** the Agent2Agent (A2A) protocol reached a Linux-Foundation-governed v1.0 with 150+ orgs in production, positioned as agent↔agent delegation layered alongside MCP's agent↔tool role. Smith's subagent/orchestrator delegation (AS-044, AS-046, AS-150) is currently Smith-to-Smith only; whether to speak A2A for cross-vendor delegation is worth a future spike once the protocol's adoption curve is clearer, but isn't concrete enough yet for a ticket.

## Research notes and implications

### Extensibility is becoming table stakes

Claude Code and Codex both support skills, while Claude also exposes subagents, hooks, MCP, plugins, and an SDK. Smith already has many equivalent primitives in the backlog or implementation, but should keep emphasizing **portable formats and open session data** rather than matching every vendor-specific setting one-for-one.

### Parallel/background work needs a control plane

OpenCode advertises parallel sessions, Cline has a Kanban/worktree story, Claude Code exposes background mode, and Codex web/cloud tasks make delegated work visible. Smith has async runner and orchestrator tickets, but lacks a concise user-facing workboard concept that unifies local TUI, headless, and daemon jobs.

### Codebase intelligence should be explicit and inspectable

Aider's repo map is a simple, memorable primitive. IDE agents also benefit from real-time editor/linter awareness. Smith's context store can support something stronger: a versioned project-intelligence map whose entries have source file citations, freshness, token-cost estimates, and inclusion/exclusion provenance.

### Sharing and collaboration are under-specified

OpenCode share links and cloud task pages set user expectations for easy debugging and handoff. Smith's read-only WASM inspector and event log are a strong base, but the product needs a safe export/share contract with redaction, TTL, and offline bundle options.

### Background cost transparency is a competitive wedge

As agents add automatic reviews, background work, and subagents, users need per-activity attribution and hard limits. Smith's budget and cost primitives should expand from foreground turns to every autonomous child task, hook, background job, and model-assisted insight pass.

### "Walk away and get told when it matters" is becoming the default UX

Ona's checkpointed cloud agents and Cursor's mobile push notifications both assume the user leaves and expects to be pulled back in only when a run finishes, fails, or needs a decision. Smith's async runner (AS-054) still requires polling `smith runs list/status`; AS-171 scopes closing that gap with a provider-neutral notification contract rather than a hosted mobile product.

### Team governance is emerging as a distinct layer from per-run controls

Coder's AI Governance add-on (centralized key gateway, prompt/tool audit trail, policy enforcement) is a team-facing concern that sits above Smith's existing per-run secret handling (AS-154) and per-run cost ledger (AS-165). AS-172 scopes whether Smith should own any part of that layer or intentionally leave it to external tooling reading the append-only event log, consistent with D9.

### Interactive terminal control is a real capability gap, not just polish

Warp open-sourcing "Full Terminal Use" (PTY-level buffer access, responding to interactive prompts mid-session) exposes a genuine gap: Smith's shell tool (AS-015) only spawns and captures. Anything that expects interactive stdin — a DB shell, a debugger, an interactive migration prompt — currently can't be driven by Smith at all. See AS-186.

### Mobile/remote control converged from three vendors within two weeks

Codex Remote reached GA, Cursor's iOS app entered public beta, and Sourcegraph Amp shipped a mobile watch/drive view — all in the same two-week window. This is no longer one vendor's mobile bet; "walk away and steer from a phone" is becoming a baseline expectation, not just "walk away and get notified" (AS-171 already covers notifications). See AS-188 for the remote-steer half of that gap.

### Convention mining from repo history is a distinct signal from live-session learning

Smith's living-skills loop (D7, AS-048) learns from a session's own trial-and-error. Qodo's Custom Rules Miner shows a second, complementary signal source: an existing repo's git/PR/review history already encodes conventions a new session shouldn't have to rediscover turn-by-turn. See AS-187.

### Agent-to-agent protocols are maturing alongside MCP

The Agent2Agent (A2A) protocol reached a Linux-Foundation-governed v1.0 with 150+ orgs in production, layered as agent↔agent delegation alongside MCP's agent↔tool role. Zed's ACP (a different, editor-hosting-focused protocol) is already on Smith's roadmap via AS-052. A2A is a distinct, later question — whether Smith's orchestrator/subagents (AS-044, AS-046, AS-150) should be able to delegate to or accept delegation from non-Smith agents — and isn't concrete enough yet to ticket; revisit once the adoption curve or a specific integration need is clearer.

### Vendor churn validates provider-neutrality as a hedge, not just a technical bet

Google discontinued Gemini CLI's free/Pro/Ultra serving and replaced it with a renamed "Antigravity CLI" within weeks, and Cognition retired Windsurf/Cascade in favor of "Devin Local." Users of both products absorbed a forced migration. D1's provider-neutrality bet protects Smith users from exactly this: no single vendor's rebrand or shutdown can strand a Smith session, since the data substrate and CLI surface are Smith's, not the provider's.
